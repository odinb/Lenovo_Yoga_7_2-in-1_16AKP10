# Lenovo Yoga 7 2-in-1 16AKP10 (83JU / LNVNB161216) — Linux Audio Setup Guide

## Prerequisites:
- Manjaro KDE installed, kernel 6.12+
- calf and lsp-plugins-lv2 installed: `sudo pacman -S calf lsp-plugins-lv2`

Step 1: Patch the HDA Codec Module<br />
The ALC287 codec on this laptop (subsystem ID 17aa:391d) is not yet recognized by the stock kernel, so volume control won't work without a patched module.

1a. Get the kernel source matching your running kernel:<br />
`uname -r` # e.g. 6.18.12-1-MANJARO<br />
`sudo pacman -S linux618-headers`<br />
`git clone --depth=1 --branch v6.18.12 \
  https://github.com/torvalds/linux.git ~/linux-source`

OR manually download just the realtek codec files from the kernel mirror

1b. Apply the patch:<br />
Edit `~/linux-source/sound/hda/codecs/realtek/alc269.c` and find the `alc269_fixup_tbl[]` array. Add this line in the 17aa: section (before the terminating entry):<br />
`cHDA_CODEC_QUIRK(0x17aa, 0x391d, "Lenovo Yoga 7 2-in-1 16AKP10", ALC287_FIXUP_YOGA9_14IAP7_BASS_SPK_PIN),`

1c. Build the module
sudo pacman -S base-devel
make -C /usr/lib/modules/$(uname -r)/build \
  M=~/linux-source/sound/hda/codecs/realtek \
  modules

1d. Install the patched module
# Compress
zstd ~/linux-source/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko

# Backup original
sudo cp /usr/lib/modules/$(uname -r)/kernel/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko.zst \
  /usr/lib/modules/$(uname -r)/kernel/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko.zst.bak

# Install patched
sudo cp ~/linux-source/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko.zst \
  /usr/lib/modules/$(uname -r)/kernel/sound/hda/codecs/realtek/

sudo depmod -a
sudo reboot

1e. Verify after reboot
sudo dmesg | grep -i "ALC287\|391d\|fixup"
# Should show: "ALC287: picked fixup for codec SSID 17aa:391d"

=====
Step 2: Fix ALSA Speaker Levels
By default the hardware speaker volume is set to 64% which sounds weak. Set to 100% and save:
amixer -c 1 sset 'Speaker' 100%
amixer -c 1 sset 'Bass Speaker' on
amixer -c 1 sset 'Master' 100%
sudo alsactl store
Note: Speaker is set to 100%, to avoid hardware clipping distortion when EQ bass boost is applied, this might need to be reduced.

=====
Step 3: PipeWire EQ Filter Chain
This adds a permanent equalizer in PipeWire — no extra apps needed, works on every boot.
3a. Create the config
mkdir -p ~/.config/pipewire/pipewire.conf.d
vi ~/.config/pipewire/pipewire.conf.d/eq-laptop.conf
```

Paste the entire contents:
```

3b. Set EQ as default sink
mkdir -p ~/.config/wireplumber/wireplumber.conf.d
vi ~/.config/wireplumber/wireplumber.conf.d/default-sink.conf
```

Paste:
```
default.configured-nodes = [
  {
    type = sink
    name = "effect_input.eq"
  }
]

3c. Restart audio stack
systemctl --user restart pipewire pipewire-pulse wireplumber

3d. Verify
pactl list sinks short
# Should show effect_input.eq as RUNNING when audio plays

wpctl status | grep -i "sink\|default"
# effect_input.eq should have * (default marker)

=====
Troubleshooting
Volume control stops working:
wpctl set-volume @DEFAULT_AUDIO_SINK@ 0.5
wpctl set-mute @DEFAULT_AUDIO_SINK@ 0

EQ not applying (audio sounds flat):
# Check EQ sink exists
pactl list sinks short
# Set it as default (use the ID shown for effect_input.eq)
wpctl set-default <ID>
# Restart Brave/Firefox to reconnect to new default sink

Speaker level reset to 64%:
amixer -c 1 sset 'Speaker' 70%
sudo alsactl store

=====
After kernel update — patch will be lost, rebuild and reinstall snd-hda-codec-alc269.ko.zst against new kernel headers. Volume control will stop working as the symptom.

Re-patching After Kernel Update:
1. Check new kernel version
uname -r
# e.g. 6.18.15-1-MANJARO

2. Install new kernel headers
# Replace 618 with your kernel series number (e.g. 619, 620)
sudo pacman -S linux618-headers

3. Verify the patch is still in your source
grep "391d" ~/linux-6.18.12/sound/hda/codecs/realtek/alc269.c
# Should show: HDA_CODEC_QUIRK(0x17aa, 0x391d, "Lenovo Yoga 7 2-in-1 16AKP10"...
If the line is there, skip step 4.

4. If source is gone or needs updating, re-apply the patch
# Find the fixup table in alc269.c and add the line back
vi ~/linux-6.18.12/sound/hda/codecs/realtek/alc269.c
# Find: SND_PCI_QUIRK(0x17aa, 0x38b5, "Legion Slim 7 16IRH8"
# Add BEFORE it:
# HDA_CODEC_QUIRK(0x17aa, 0x391d, "Lenovo Yoga 7 2-in-1 16AKP10", ALC287_FIXUP_YOGA9_14IAP7_BASS_SPK_PIN),

5. Clean old build artifacts
cd ~/linux-6.18.12/sound/hda/codecs/realtek/
make -C /usr/lib/modules/$(uname -r)/build \
  M=~/linux-6.18.12/sound/hda/codecs/realtek \
  clean

6. Rebuild against new kernel headers
make -C /usr/lib/modules/$(uname -r)/build \
  M=~/linux-6.18.12/sound/hda/codecs/realtek \
  modules

7. Compress, backup and install
# Compress
zstd ~/linux-6.18.12/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko

# Backup new stock module
sudo cp /usr/lib/modules/$(uname -r)/kernel/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko.zst \
  /usr/lib/modules/$(uname -r)/kernel/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko.zst.bak

# Install patched module
sudo cp ~/linux-6.18.12/sound/hda/codecs/realtek/snd-hda-codec-alc269.ko.zst \
  /usr/lib/modules/$(uname -r)/kernel/sound/hda/codecs/realtek/

sudo depmod -a
sudo reboot

8. Verify after reboot
sudo dmesg | grep -i "ALC287\|391d\|fixup"
# Must show: "ALC287: picked fixup for codec SSID 17aa:391d"
# If it shows 38b5 instead, the patch didn't take — check step 6

Note: If Manjaro updates to a new kernel series (e.g. 6.19), you'll need to install linux619-headers and the new kernel source may need the patch re-applied to a fresh copy of alc269.c. The patch line itself won't change — only the kernel version you build against changes.
Tip: Once this patch lands officially in the upstream kernel (Bug 221210), none of this will be needed anymore — the fix will be included automatically in every kernel update.

A few ways to track it:
1. Watch the kernel bugzilla directly
https://bugzilla.kernel.org/show_bug.cgi?id=221210
Click "Add me to CC list" at the bottom — you'll get email when there's any activity, comments, or status changes.

2. Watch the upstream kernel git for the fix
# Once you think it might be merged, search the kernel changelog
curl -s https://raw.githubusercontent.com/torvalds/linux/master/sound/hda/codecs/realtek/alc269.c | grep "391d"
When that returns your line, it's in mainline.

3. Check when Manjaro picks it up
# After a kernel update, just check dmesg - if it still works without your patched module:
sudo dmesg | grep "391d"
# If it shows up WITHOUT you having installed the patched module, it landed

4. Simplest ongoing check after any kernel update
# Before repatching, test if it's needed:
sudo dmesg | grep -i "fixup"
# If it shows 17aa:391d naturally → patch is in kernel, you're done forever
# If it shows 17aa:38b5 → still need to repatch

Realistically, given that Zhang Heng submitted the patch and it's already in review on the bug, it will likely land in kernel 6.13 or 6.14 and flow into Manjaro within a few weeks of that.
